# 5.0 - Lógica de Controle de Ações nos Cards (UI)

## Objetivo
- Atualizar o frontend para refletir os novos status e controlar quais ações (Cadastrar, Analisar, Cancelar) o usuário pode realizar.

## Escopo / Entregáveis
- Alteração no arquivo JS responsável pela renderização dos cards (ex: `public/js/similaridade.js`).
- Lógica de habilitação/desabilitação de botões e badges de status.

## Passos e subtarefas
- 5.1 Atualizar a lógica de renderização para tratar o status `cadastrado_sem_prazo`.
- 5.2 Implementar regra: Se status for `cadastrado_sem_prazo`, desabilitar "Cadastrar" e "Cancelar", habilitar "Analisar".
- 5.3 Implementar regra: Se status for `pendente`, habilitar "Cadastrar" e "Cancelar", desabilitar "Analisar".
- 5.4 Adicionar feedback visual (ex: badge ou cor de fundo diferenciada) para itens cadastrados aguardando análise.

## Dependências
- 3.0 (Backend de Cadastro).

## Paralelizável?
- Sim, com a tarefa 4.0.

## Critérios de aceite
- O usuário não consegue cadastrar um item duas vezes.
- O botão "Analisar" só fica clicável após o item ser cadastrado.
- O botão "Cancelar" fica inacessível após o cadastro.

## Testes
- 5.5 Validar visualmente a mudança de estado dos botões após o clique em "Cadastrar".
- 5.6 Verificar se a UI carrega o estado correto após um refresh da página.

## Notas
- Considerar o uso de classes CSS para indicar o estado desabilitado.
